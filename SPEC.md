# Delivery — спецификация

Внутренняя система учёта посылок, которые едут с eBay в США через посредников в Кыргызстане до конечного получателя в России. Документ описывает целевое состояние продукта и его бизнес-логику. Технические детали (схема БД, конкретные эндпоинты) описаны словами без кода — чтобы по тексту можно было построить как реализацию, так и понимание целиком.

---

## 1. Зачем это

Степан заказывает запчасти и товары на eBay. Покупки приходят на адрес склада в США, который держит знакомая команда в Кыргызстане. Они получают посылки, периодически собирают их в общую партию (фурой или курьерской службой) и отправляют в Россию.

Раньше учёта этого процесса не было. Невозможно было понять:
- какие из последних заказов уже физически приехали в США, а какие застряли в пути;
- что лежит на складе в КГ и ждёт следующей отправки в Россию;
- кто из посылок уехал в Россию и в какой партии;
- какая сумма должна команде в Кыргызстане за переправку (вес × тариф);
- какие посылки давно должны были приехать, но не приехали — то есть «горящие».

Эта система решает обе стороны проблемы: даёт менеджеру полную картину «где что находится и сколько я должен», а команде форвардеров — простой инструмент для отметки этапов и формирования отгрузок в Россию.

---

## 2. Кто пользуется

Пользователей в системе ровно два типа.

**Менеджер** — это Степан, заказчик. Один общий логин. Видит всё содержимое заказов, цены, продавцов eBay, суммы к оплате форвардерам. Может настраивать тариф и пороги «горения». Принимает партии, когда они физически приезжают в Россию: открывает страницу прибывшей отгрузки и по каждому треку отмечает «получено» или «не получено». В UI слово «админ» не используется — везде «менеджер».

**Форвардер** — это команда в Кыргызстане. Один общий логин. Получает посылки в США (опционально), отмечает приход в КГ, формирует и отправляет отгрузки в Россию.

Структура поддерживает добавление дополнительных аккаунтов в обеих ролях, но на старте — ровно два общих логина. Аутентификация по логину и паролю в открытом виде из переменных окружения (constant-time сравнение через `hmac.compare_digest`); хеши не используем сознательно — это внутренний инструмент с двумя пользователями, и хеш только усложняет ротацию.

## 3. Что видит и чего не видит каждая роль

Это критичный раздел. Форвардер не должен видеть **ничего**, что касается содержимого заказов и денежных сумм. Это требование приватности и бизнес-условие одновременно: форвардер физически работает с посылками, а не с информацией о товарах внутри.

**Форвардер видит:** трек-номер, текущий статус, плановые и фактические даты этапов, опциональный комментарий менеджера, вес посылки, фото посылки, отгрузку в которую попала посылка.

**Форвардер не видит:** название товара, продавца на eBay, номер заказа на eBay, цену товара, сумму налога, сумму доставки до США, сумму к оплате за переправку (вес × тариф), идентификаторы заказа в источнике.

**Менеджер видит всё то же** плюс все скрытые от форвардера поля. Дополнительно — суммарную задолженность форвардерам (по всем посылкам в активных отгрузках КГ→РФ по текущему тарифу), активные отгрузки с типом транспорта, историю тарифов, отдельную страницу заказов без трека.

Это разделение обеспечивается на уровне API: для роли «форвардер» серверный слой возвращает урезанную модель посылки, без скрытых полей. Это техническая гарантия — даже если на фронте случайно отрендерить такое поле, в данных его не будет.

**Excel-экспорт даёт обоим ролям одинаковый набор колонок без приватных полей** (трек, статус, даты, вес, заметка). Это специально: файл может быть случайно переслан куда угодно, и в нём не должно оказаться ни цен, ни названий товаров.

## 4. Откуда берутся заказы

Источник данных — отдельная база `ebay_orders`, которая наполняется парсером eBay (живёт независимо от этой системы). Она содержит таблицу заказов с полями: номер заказа, продавец, дата заказа, общая сумма, статус доставки, дата доставки, ETA, признак untracked. К каждому заказу могут быть привязаны один или несколько трек-номеров (таблица `order_tracking_numbers`).

Наша система **не модифицирует** источник. Мы только читаем оттуда. Технически — через PostgreSQL Foreign Data Wrapper, который позволяет нашей базе обращаться к таблицам внешней БД как к своим (схема `ebay_remote`). Синхронизация работает по расписанию: каждые несколько минут (по умолчанию 5) фоновый воркер берёт из источника заказы (с учётом cutoff-даты) и синхронизирует их со своими данными.

**Важные правила синхронизации:**

- **Главная сущность — трек-номер, не заказ.** Если у одного eBay-заказа два трека, мы создаём две независимые посылки. Дальнейшая жизнь у них разная: каждая может попасть в свою отгрузку, прийти в разные дни, потеряться по отдельности.

- **Что воркер пишет при первом импорте трека (вставка):** базовые поля из источника + статус. Если в источнике уже есть `delivered_date` (то есть трек уже физически в США) — для имитации «реалистичных данных» статус заполняется случайно из множества `arrived_usa / received_by_forwarder_usa / arrived_kg / in_shipment_kg_to_ru / delivered_ru` с весовым распределением. Это сделано сознательно по запросу владельца («раскидаем рандомно ради моков, потом данные поменяю»). Если `delivered_date` пуст — статус `ordered`. Эта рандомизация работает **только при создании** записи; уже существующие записи воркер не перезатирает.

- **Что воркер обновляет при повторных проходах:** только справочные поля из источника — продавец, название товара, общая сумма, ETA, фактическая дата прихода в США (если появилась). **Статус, вес, привязки к отгрузкам не трогает** — это поля, которыми владеет наша система, а не источник.

- **Cancelled-заказы.** Если в `delivery_status` источника встречается слово «cancel», такая пара (заказ, трек) пропускается при первом импорте — мы не создаём для них посылку. Если посылка к моменту отмены уже физически где-то находится, статус не меняется автоматически — менеджер увидит и решит вручную.

- **Untracked-заказы** (без трек-номеров или с флагом `is_untracked=true`) в основной список не попадают. У менеджера есть отдельная страница «Без трека», где они видны справочно (берутся проекцией прямо через FDW, своих копий не делаем). Если у такого заказа потом появится трек, он автоматически уйдёт в основной поток на следующем тике синка.

- **Cutoff-дата.** На старте импортируем все заказы (по умолчанию `1900-01-01`, то есть фактически без отсечки). Можно поменять в настройках, чтобы синк начал брать только заказы от выбранной даты.

- **ETA-парсинг.** В источнике `arriving_by_date` хранится строкой в форматах: `Fri, May 22`, `May 12, 2026`, `May 1, 2026 - May 12, 2026`. Парсер собирает дату по правилам: при отсутствии года берётся ближайший в будущем относительно даты заказа; день недели валидируется (несовпадение — в лог); из диапазона берётся **верхняя** граница. Если строку не удалось разобрать — ETA остаётся пустым, в интерфейсе пишется «дата прибытия неизвестна».

## 5. Жизненный цикл трека и переходы статусов

Жизненный цикл одной посылки от заказа до доставки в Россию проходит через определённые этапы. У каждого этапа точные правила того, кто и как его выставляет.

**Этап 1: «Заказано» (`ordered`).** Появляется автоматически, когда синхронизация подхватывает новый заказ из источника. Видим у форвардера сразу. ETA считаем по `arriving_by_date`.

**Этап 2: «Прибыл в США» (`arrived_usa`).** **Только автомат**: ставится при наличии `delivered_date` в источнике (eBay подтвердил доставку). Форвардеру выставлять руками запрещено.

**Этап 3: «Получен в США» (`received_by_forwarder_usa`).** Опциональное действие форвардера. Когда команда физически забрала посылку со склада. Можно пропустить.

**Этап 4: «В пути в КГ» (`in_shipment_usa_to_kg`).** Опциональный статус. Отдельного UI для US→KG отгрузок нет — форвардер при необходимости ставит этот статус сам через bulk-действие, или треки сразу прыгают в `arrived_kg` минуя этот этап.

**Этап 5: «Прибыл в КГ» (`arrived_kg`).** Обязательный статус. Форвардер ставит, когда посылка физически на складе в Бишкеке. Опционально вводит вес и грузит фото. Без веса этот статус разрешён, но без веса посылку нельзя будет отправить дальше в Россию.

**Этап 6: «В пути в Россию» (`in_shipment_kg_to_ru`).** **Только автомат**: ставится в момент, когда форвардер нажимает «Отправить отгрузку» (направление KG→RU). Никто не выставляет руками — этот переход всегда побочный эффект отправки отгрузки.

**Этап 7: «Доставлено в Россию» (`delivered_ru`).** Финальный успешный статус. Ставится менеджером при приёме отгрузки. Альтернативный финал — `not_received_ru` (посылка была в отгрузке, но при разборе её не нашли) с автоматическим флагом проблемы `lost`.

**Что разрешено форвардеру в bulk-действиях** (выбрал треки → «Сменить статус»):
- `received_by_forwarder_usa` — опц.
- `in_shipment_usa_to_kg` — опц.
- `arrived_kg` — обяз.

Форвардер может «прыгнуть» в любой из этих трёх статусов из любого предыдущего нетерминального состояния (в том числе сразу из `ordered` в `arrived_kg`, если посылка приехала так быстро, что промежуточные статусы не успели зафиксироваться).

Запрещено форвардеру: `arrived_usa` (автомат), `in_shipment_kg_to_ru` (только через отправку отгрузки), `delivered_ru` / `not_received_ru` (только менеджер при приёмке), `cancelled`. Все эти запреты валидируются на бэке: даже если форвардер обойдёт UI и пошлёт PATCH вручную — придёт `403 forbidden_transition:from->to`.

**Что разрешено менеджеру:** всё. Откаты статуса назад работают, но в UI на них стоит подтверждение в диалоге.

**Импорт xlsx уважает те же правила.** Если форвардер попытается через xlsx откатить трек или поставить запретный статус — строка попадёт в «ошибки» с понятной причиной (`forbidden_transition:from->to`) и не применится. Менеджеру всё разрешено.

**На любом этапе** к посылке может быть применён флаг проблемы: `lost`, `damaged`, `delayed`. Эти флаги отдельны от основного статуса, используются для пометки вручную.

Все переходы фиксируются в `parcel_history` — кто, когда, из какого статуса в какой. Это и аудит, и расчёт длительности этапов на странице трека.

## 6. Отгрузки

Отгрузка — это партия посылок, которая едет одним транспортом из одного места в другое. Активно используется одно направление — Кыргызстан → Россия. Направление США→Кыргызстан в схеме поддержано, но отдельный пользовательский поток для него на старте не создаётся (статус `in_shipment_usa_to_kg` ставится форвардером вручную, если нужно).

**Жизненный цикл отгрузки:** черновик (`draft`) → в пути (`in_transit`) → принята (`received`).

### Черновик

«Копилка», в которую форвардер набирает треки. Создаётся пустой через кнопку «Новая отгрузка». При создании форвардер заполняет (всё опционально):

- **Транспорт** — выбор из быстрых вариантов «Фура» / «СДЭК» либо своё значение. Поезд и самолёт исключены.
- **Планируемая дата отправки** / **планируемая дата прибытия** — опц.
- **Номер накладной** — текст.
- **Фото накладной** — изображение или PDF, грузится в MinIO-бакет `delivery-photos` под ключом `waybills/{shipmentId}/...`.
- **Заметка** — свободный комментарий.

Все поля черновика **редактируются в любой момент, пока отгрузка не отправлена**. После отправки — поля становятся read-only.

**Несколько черновиков параллельно — это норма.** Форвардер может вести одновременно «Фура на следующую неделю» и «СДЭК срочно».

#### Добавление треков в черновик

- **Только треки в статусе `arrived_kg`** могут попасть в KG→RU отгрузку. Это валидируется на бэке: если послать трек с любым другим статусом — придёт `409 parcel_must_be_in_kg:got_<status>`.
- **Только треки, ещё не привязанные к другой отгрузке.** Повторное добавление возвращает `409 parcel_already_in_shipment`.
- Способов добавить два:
  - **Со страницы конкретного трека** (когда он в КГ): кнопка «Добавить в отгрузку» открывает список открытых черновиков.
  - **Со страницы самого черновика**: блок «Готовы к отправке в КГ» внизу. Видны все треки в КГ, не привязанные ни к какой отгрузке. У каждого — поиск по подстроке трека, кнопка «Добавить» справа.

**Удалить трек из черновика** можно так же легко (иконка корзины в строке), пока отгрузка не отправлена.

#### Список треков в черновике

Каждая строка в блоке «В этой отгрузке»:

- **Трек-номер** — кликабельный, ведёт на страницу детали трека (`/forwarder/track/{tn}`).
- **Иконка copy** — рядом с треком, копирует трек-номер в буфер.
- **Inline-input веса** — заменяет статичную надпись «? кг» в черновике. Форвардер вводит число прямо в строке, на blur (или Enter) выполняется PATCH. Запятая принимается как разделитель. Сохранение не блокирует ввод соседних строк — каждая строка независима.
- **Иконка корзины** — убрать трек из черновика.

После того как отгрузка отправлена, inline-input исчезает, остаётся текстом «N кг», корзина исчезает.

#### Excel в черновике

- **«Скачать xlsx»** — выгружает все треки этой отгрузки.
- **«Импорт xlsx»** — двухшаговый: загрузить → превью (что обновится / что пропустится / какие ошибки) → применить. Типичный сценарий: партия пришла на склад, ребята взвесили на бумаге, вбили в Excel, грузанули — система проставила веса всем разом.

После отправки отгрузки кнопка «Скачать xlsx» работает, «Импорт xlsx» исчезает (нечего править).

### Отправка черновика

Когда машина реально выезжает, форвардер нажимает «Отправить отгрузку». В этот момент:

- Черновик переходит в статус `in_transit`, фиксируется фактическая дата отправки.
- Все треки переходят в статус `in_shipment_kg_to_ru` (для направления KG→RU).
- Для каждого трека фиксируется снимок цены: `вес × текущий_тариф`, в долларах. Снимок не меняется, даже если позже тариф изменится.
- В `parcel_history` записывается переход.

**Отправка с пустым черновиком невозможна** — `409 empty_shipment`. UI показывает понятное сообщение «В отгрузке нет ни одного трека».

**Отправка с треками без веса невозможна** — `409 missing_weight:tn1,tn2,...`. UI обрабатывает это специально:

- редирект остаётся на странице отгрузки, в URL добавляется `?missing=tn1,tn2,...`;
- внизу появляется красный toast: «Нельзя отправить: у N треков не указан вес» + перечень треков;
- проблемные строки в списке подсвечиваются красным фоном, рамка inline-input'а становится красной;
- форвардер вводит веса прямо на месте, нажимает «Отправить» снова.

### После отправки (`in_transit` / `received`)

- **Поля отгрузки read-only** — нельзя менять транспорт, накладную, фото, даты, заметку.
- **Треки read-only** — нельзя добавить новые, нельзя удалить из отгрузки, нельзя сменить им статус через bulk-действия (физически они в кузове).
- **«Скачать xlsx» работает**, **«Импорт xlsx» — нет** (нечего править, посылки уже едут).

### Приём отгрузки в России

Когда машина приехала и груз разобран, менеджер открывает страницу приёма этой отгрузки. По каждому треку проставляет «получено» или «не получено». Доступно массовое действие: выделить чекбоксами и нажать «Получено» / «Не получено». После завершения приёма отгрузка переходит в `received`, все полученные треки — в `delivered_ru`, непринятые — в `not_received_ru` с флагом проблемы `lost`.

## 7. Тариф и оплата

Тариф — цена за килограмм веса, по которой менеджер оплачивает услугу форвардеру за переправку из КГ в Россию. Тариф один общий, не зависит от типа транспорта или объёма. Текущее значение — 9.5 долларов за килограмм.

Тариф **историчен**: хранится не одно текущее значение, а вся история в таблице `tariffs`. Старые отгрузки сохраняют ту цену, по которой были созданы. Если менеджер поменяет тариф с 9.5 на 10 — это повлияет только на отгрузки, которые будут созданы после изменения.

**Снимок цены** для конкретной посылки фиксируется в момент перехода в `in_shipment_kg_to_ru` — то есть когда форвардер нажимает «Отправить» на черновике отгрузки. В этот момент берётся вес посылки, текущий тариф, и в посылку записывается результат умножения. Дальше эта сумма не меняется.

**Кто видит сумму:** только менеджер. Форвардер сумму **не видит**, только вес и трек. Это сделано намеренно: операционно ему не нужна эта информация, а коммерческие условия — между ним и менеджером, не часть рабочего инструмента.

Изменение тарифа делается на странице настроек менеджером. Старое значение остаётся в истории.

## 8. Просрочки — «долго не доезжает»

Система отдельно отслеживает посылки, которые слишком долго находятся на одном этапе. Это критичный механизм — без него теряются посылки, которые «зависли» где-то в пути и о них все забыли.

Логика по этапам:
- **Долго не доезжает в США** — статус `ordered`, и со дня предполагаемого прибытия (ETA) прошло больше N дней.
- **Долго едет в Кыргызстан** — статус `in_shipment_usa_to_kg`, и со дня попадания в эту отгрузку прошло больше M дней.
- **Долго едет в Россию** — статус `in_shipment_kg_to_ru`, и со дня отправки прошло больше K дней.

N, M, K — настраиваемы в настройках отдельно для каждого этапа. По умолчанию: США 7 дней, США→КГ 21 день, КГ→РФ 7 дней. У каждого порога есть переключатель «учитывать ли вообще» — можно временно отключить, не сбрасывая значения.

**Где видно:**
- Сверху главной страницы и у менеджера, и у форвардера — красная панель «Долго не доезжает · N» со списком проблемных треков. Это первое что видит человек, открывая систему.
- В таблице треков ряды-просрочки имеют светло-красный фон.
- В колонке «Дата» вместо обычной даты пишется человеческая фраза: «долго не доезжает в США», «долго едет в Кыргызстан», «долго едет в Россию».

Сознательно используется мягкая фраза «долго не доезжает» вместо «просрочено» — задержка не всегда означает реальную проблему (бывают объективные задержки на таможне), но требует внимания.

## 9. Excel: импорт и экспорт

Базовый рабочий инструмент для массовых операций. Нужен и менеджеру, и форвардеру. Колонки одинаковые для обеих ролей (см. п. 3) — без цен, продавцов, названий товаров.

**Колонки:** `Трек`, `Статус`, `Заказан`, `Прибыл в США`, `Получен в США`, `Прибыл в КГ`, `Отправлен в РФ`, `Доставлен в РФ`, `Вес, кг`, `Заметка`.

### Экспорт

- **С главной** (треки): кнопка «Скачать xlsx» в bulk-плашке, когда выделены треки. Если ничего не выделено — выгрузка идёт через выделение всех треков фильтра.
- **Со страницы отгрузки**: кнопка «Скачать xlsx» — выгружает треки этой отгрузки. Доступна и в черновике, и в `in_transit`/`received`.
- Файл всегда содержит: замороженную первую строку (header), autofilter, **dropdown-валидацию на колонке `Статус`** (Excel сам не пустит ввести левое значение), имя файла `delivery-tracks-YYYY-MM-DD.xlsx` или `delivery-shipment-{id}-YYYY-MM-DD.xlsx`.

### Импорт

Один универсальный поток:

1. Кнопка «Импорт xlsx» (на главной, на странице draft-отгрузки).
2. Выбор файла → POST на бэк → возвращается **превью**: сколько строк обновится, сколько без изменений, сколько ошибок (с причинами по каждой строке).
3. Превью показывает по каждой обновляемой строке — какие именно поля изменятся (`вес: 1.85 → 2.10`, `статус: ordered → arrived_kg`).
4. Кнопка «Применить N» применяет валидные строки.

**Что считается ошибкой и пропускается** (никогда не применяется тихо):
- Несуществующий трек (`not_in_system`).
- Невалидный вес (`invalid_weight:abc`).
- Запрещённый переход статуса для этой роли (`forbidden_transition:ordered->in_shipment_kg_to_ru`).
- Неизвестное значение в колонке статус (`unknown_status:xxx`).

**Колонки читаются по названию, не по позиции** — если пользователь переставил столбцы, импорт всё равно работает.

## 10. UX-принципы

**Один паттерн для обеих ролей.** Никаких боковых панелей и нижних bar-меню. Везде сверху одна горизонтальная шапка: бренд слева, навигация по центру, поиск + переключатель темы + имя пользователя + кнопка выхода справа.

**Шапка содержит только навигацию и общие действия.** Кнопки «Импорт xlsx» и «Новая отгрузка» **не в шапке**, а на главной странице — отдельным блоком под красной панелью «долго не доезжает» и до «Открытые отгрузки». Логика: красная плашка — самое горящее, кнопки — что делать дальше, потом списки.

**Поиск.** Иконка лупы в шапке у обеих ролей. По клику разворачивается поле, ввод фильтрует. Форвардер ищет по трек-номеру (substring). Менеджер — по трек-номеру, названию товара и номеру заказа в eBay. Состояние поиска — в URL, можно поделиться ссылкой.

**Трек-номер в любом списке = ссылка на страницу трека.** Кликабелен сам номер. Рядом — отдельная маленькая иконка copy для копирования в буфер обмена (показывает галочку «скопировано» на ~1 секунду). Это единый паттерн на всех экранах: главные списки, страница отгрузки, страница приёма.

**Toast-сообщения для обратной связи.** Все ошибки операций (от 409 API до невалидного состояния) показываются единым компонентом toast'а внизу экрана: иконка alert + понятный заголовок + детали + крестик закрытия (авто-закрытие через 8 секунд). Никаких голых кодов ошибок и Runtime Error экранов — server actions ловят `ApiError` и редиректят на ту же страницу с `?error=<код>`; страница превращает код в человеческое сообщение по словарю.

**Светлая и тёмная темы.** Переключатель в шапке, выбор сохраняется в localStorage. По умолчанию — тёмная. Переключение реализовано через CSS-переменные на корневом элементе.

**Списки, не карточки.** Треки показываются строками таблицы/списка, а не плиточным гридом. Карточки остаются только для отдельных сущностей с богатым содержимым (открытые отгрузки на главной форвардера).

**Просроченные подсвечены красным фоном** в строках таблицы. Без отдельных бейджей — менее визуально шумно.

**Никаких англицизмов.** «ETA» → «придёт 13 мая», «Dashboard» → «Главная», «admin» → «менеджер». Всё на русском, в обычном письменном стиле.

**Заголовки в editorial-стиле.** Большой serif (Fraunces), сентенс-кейс с точкой в конце: «Главная.», «Треки.», «Новая отгрузка.».

## 11. Выборка треков и массовые действия

Чекбокс есть **в каждой строке** списка треков (и у менеджера, и у форвардера). Механика одинаковая, отличается только набор доступных действий по роли.

**«Выбрать все»** — отдельный header-чекбокс над списком (с поддержкой indeterminate-состояния).

При выборе хотя бы одного трека внизу появляется **sticky bulk-плашка** с кнопками. Состав кнопок зависит от экрана:

| Экран | Кнопки в плашке |
|---|---|
| Главная (менеджер) | Скачать xlsx · Сменить статус · Снять |
| Главная (форвардер) | Скачать xlsx · Сменить статус · В отгрузку · Снять |
| Страница draft-отгрузки | Скачать xlsx · Убрать из отгрузки · Снять |
| Страница приёма (менеджер) | Скачать xlsx · Получено · Не получено · Снять |
| Страница in_transit / received отгрузки | Скачать xlsx · Снять (всё read-only) |

«Сменить статус» открывает диалог с кнопками — каждая кнопка это допустимый целевой статус для роли (см. п. 5). После применения показывается отчёт: сколько применено, какие пропущены и почему. Невалидные строки **никогда не применяются молча**.

«В отгрузку» (форвардер) открывает список открытых черновиков — ровно один клик чтобы добавить выбранные треки в выбранный черновик. Бэк проверит, что у каждого выбранного трека статус `arrived_kg` (см. п. 6), и пропустит непригодные.

## 12. Адаптивность

Система рассчитана на работу и с телефона (форвардер на складе с мобильного), и с большого экрана (менеджер за компьютером).

- **Десктоп (от 1080px).** Полная ширина, 5 колонок в таблице треков менеджера, счётчики в три колонки. Строка треков у форвардера — 5 колонок: чекбокс · трек · статус-чип · дата · шеврон.
- **Планшет (640–900px).** Колонка «оплата» в таблице менеджера скрывается, счётчики в две колонки.
- **Мобильный (до 640px).** Каждая строка списка треков — трёхстрочная: сверху трек-номер, ниже статус-чип, ниже дата (не помещаются в одну строку из-за длинных названий). Чекбокс остаётся слева на всю высоту, шеврон справа на всю высоту. Колонка «содержимое» в таблице менеджера на мобильном скрывается — её можно увидеть в детали трека. Bulk-плашка превращается в широкий блок снизу. Поиск на всю ширину. Имя пользователя в шапке скрывается, остаются иконки.

**Топ-меню никогда не превращается в гамбургер.** Пунктов навигации мало (у форвардера 2, у менеджера 3) — умещаются в строку. Это сознательное решение: гамбургеры замедляют работу и скрывают важные действия.

## 13. Данные

**Пользователь.** Идентификатор, роль (`admin` / `forwarder`), отображаемое имя. Учётные данные хранятся в env, не в таблице.

**Сессия.** UUID, ссылка на пользователя, момент создания, момент последней активности, момент истечения, user-agent, IP. Cookie `delivery_session` (HttpOnly, SameSite=Lax) с UUID. TTL — 30 дней rolling: каждый запрос продлевает срок жизни.

**Посылка.** Главная сущность. Идентифицируется трек-номером (первичный ключ). Поля:

- Текущий статус (один из этапов конвейера).
- Опциональный флаг проблемы.
- Дата заказа на eBay.
- Плановая дата прибытия в США (ETA, может быть пустой).
- Фактическая дата прибытия в США (заполняется автоматически из источника).
- Дата подтверждения получения форвардером в США (опционально).
- Ссылка на отгрузку США→КГ (опционально).
- Дата прибытия в КГ.
- Вес в килограммах (опционально до момента отправки в Россию).
- Ссылка на отгрузку КГ→РФ.
- Дата доставки в Россию.
- Список фотографий (через отдельную таблицу `parcel_photos`).
- Комментарий менеджера (опционально).
- Скрытый блок «admin only»: номер заказа eBay, продавец, название товара, общая сумма заказа, снимок тарифа, рассчитанная стоимость доставки.

**Отгрузка.** Партия. Поля: id, направление, статус, тип транспорта, список трек-номеров (через FK с posylki), дата создания, плановая/фактическая дата отправки, плановая/фактическая дата прибытия, номер накладной, ссылка на фото накладной, заметка.

**Тариф.** Историческая запись цены за килограмм: дата начала действия, цена в долларах. Новая запись — при смене.

**Настройки.** Один ряд в таблице `settings`. Содержит: тариф, дата начала действия тарифа, три порога просрочки + их вкл/выкл флаги, cutoff-дата для синка.

**История изменений.** Каждый переход статуса — отдельная строка: посылка, старый статус, новый статус, кто инициировал, заметка (например, «xlsx import», «shipment sh_004 sent»), время.

**Фото посылок.** Метаданные: ссылка на посылку, ключ объекта в MinIO, публичная ссылка, MIME-тип, размер, кто загрузил, когда. Сами файлы в бакете **`parts-photos`** под префиксом `delivery/parcels/{trackingNumber}/` (бакет общий с другим проектом, но префикс изолирует наши данные).

**Фото накладных.** Аналогично, но в бакете **`delivery-photos`** под префиксом `waybills/{shipmentId}/`.

**Заказы без трека.** Не отдельная таблица — проекция через FDW на `ebay_remote.orders` где `is_untracked=true` или нет связанных трек-номеров.

## 14. Технологии

**Источник данных.** PostgreSQL с базой `ebay_orders`, наполняется внешним парсером eBay. Подключаем через `postgres_fdw` (схема `ebay_remote`). Read-only.

**Наша база.** Отдельный экземпляр PostgreSQL 18, БД `delivery`. Содержит свои таблицы: `users`, `sessions`, `settings`, `tariffs`, `shipments`, `parcels`, `parcel_history`, `parcel_photos`, `shipment_waybills`. Расширения: `pgcrypto`, `postgres_fdw`.

**Backend.** FastAPI (Python 3.13) + asyncpg, без ORM. Pydantic 2 для валидации. MinIO Python SDK для фотографий. openpyxl для xlsx (с `DataValidation` на колонке статуса). Аутентификация — серверная сессия в Postgres (cookie HttpOnly с UUID), пароли в env в открытом виде, сравнение через `hmac.compare_digest`.

**Sync-воркер.** Отдельный python-процесс с `asyncio` loop. Раз в `SYNC_INTERVAL_MINUTES` (по умолчанию 5) делает один проход по `ebay_remote.orders` join `order_tracking_numbers`. Идемпотентный.

**Frontend.** Next.js 15 (App Router) + React 19, без UI-библиотек. Дизайн-токены и стили — собственные, в духе warm-editorial. Серверные компоненты делают рендер на сервере и зовут API через серверный fetch (cookie прокидывается через `next/headers`). Клиентские компоненты — для интерактива (поиск, тема, копирование, чекбоксы выборки, bulk-плашка, модалки, file upload, inline-input веса, toast).

**Фронтовые route handlers** проксируют Excel-операции (`/api/export.xlsx`, `/api/import/preview`, `/api/import/apply`) в FastAPI с автоматической передачей куки — чтобы браузер мог сохранить blob и не упирался в CORS.

**Хранение файлов.** MinIO. Креды в env. Бакеты:
- `delivery-photos` — public-read, для накладных. Префикс `waybills/{shipmentId}/`.
- `parts-photos` — public-read, общий с проектом parts_photos. Наши данные под префиксом `delivery/parcels/{trackingNumber}/`.

**Деплой.** Все компоненты в общей docker-сети `db_default` (как у соседнего проекта `parts_photos`). Контейнеры: `delivery-api` (FastAPI), `delivery-web` (Next.js), `delivery-sync` (воркер). Postgres и MinIO внешние, к ним подключаемся по адресу. CI/CD — GitHub Actions: пуш в main → сборка в Docker Build Cloud → публикация в GitHub Container Registry → SSH на сервер → `docker compose pull && up -d`.
