# Архитектура треков — техническая спека

Единый источник правды для трек-номеров между **ebay_orders** и **delivery**.
Здесь — данные, инварианты, алгоритмы и аудит. Продуктовая логика «незнакомых
треков» (заявки форвардера, AI-чат, плашки подтверждения) описана отдельно в
`SPEC_UNKNOWN_TRACKS.md`; здесь — только то, как треки хранятся и связываются.

Репозитории:
- ebay_orders — `/Users/stepan/Desktop/projects/ebay_orders`
- delivery — `/Users/stepan/Desktop/projects/delivery` (этот)

## 1. Доменная модель

### 1.1. Что есть трек и заказ

- В **ebay_orders** треки лежат в таблице `order_tracking_numbers(order_id, tracking_number)`
  — это связь многие-ко-многим: один трек может висеть на нескольких заказах,
  у заказа может быть несколько треков.
- На стороне eBay один заказ имеет **≤1** «родного» трека, а один такой трек —
  один или несколько заказов (консолидированная отправка: 7 кейсов в проде, до
  4 заказов на трек). Запись в ebay_orders делает **всегда чат-агент** (vision
  лишь готовит предварительные транскрипты и в чистовые таблицы не пишет).
- **Дополнительные** треки к уже существующим заказам добавляет **только delivery**
  (привязка незнакомого физического трека). Из ebay_orders дополнительный трек
  добавить нельзя — агент при сохранении заказа кладёт ровно один родной трек.

### 1.2. Группа и посылка

- **Группа заказов** — множество заказов, которые физически едут одной коробкой.
  Группа задаётся общим треком: заказы, разделяющие один трек, — одна группа.
- **Канонический трек группы** — её родной eBay-трек (`source='ebay'`). Это
  стабильный ключ посылки на всё время её жизни.
- **Алиас** — наш `source='delivery'`-трек, добавленный к группе, у которой уже
  есть канон. Это **тот же** физический парсел под другим номером.
- **Посылка** в delivery (`parcels_mutations`) ключится **каноническим треком**.
  Алиас своей строки `parcels_mutations` не создаёт. Все номера группы (канон +
  алиасы) показываются на карточке и участвуют в поиске.

Правило привязки: если канон висел на 5 заказах, новый delivery-трек
привязывается **ко всем тем же 5 заказам** (5 строк `(order_id, T_new)`).

### 1.3. Граничный случай — группа без канона

Если заказы были `is_untracked` (родного eBay-трека нет) и delivery привязывает
к ним трек, канона не существует. Тогда **первый** delivery-трек становится
каноном этой группы, и посылка (`parcels_mutations`) заводится по нему. Если
позже eBay покажет свой трек — он добавляется как обычный трек, но **ключ
посылки не переключаем** (стабильность важнее «правильности» канона).

## 2. Принципиальное решение

Единая таблица `ebay_orders.order_tracking_numbers` — единственный источник
правды для всех треков. Источник записи различаем колонкой `source`:

```
order_tracking_numbers (
    order_id         bigint  -- FK orders(order_id) ON DELETE CASCADE
    tracking_number  text
    source           text    -- 'ebay' | 'delivery'
    added_by         text    -- логин, кто записал (для 'delivery'; для 'ebay' null)
    added_at         timestamptz default now()
    PRIMARY KEY (order_id, tracking_number)
)
```

Колонок `delivery_extra_tracks` / `_meta` не существует — дубликат хранения убран.
Локальный флаг `is_manual` в delivery не нужен: «наш трек» = `source='delivery'`.

**Источники только два:** `'ebay'` (всё, что записано на стороне ebay_orders —
там всегда пишет агент) и `'delivery'` (наши дополнительные треки). Так правило
«доп.треки добавляет только delivery» наблюдается прямо в данных.

## 3. Аудит «кто записал трек»

Требование: видно, кто именно сохранил трек.

- Колонки `source` + `added_by` + `added_at` в самой строке — это первичный аудит.
- Триггер `audit_trg` в ebay_orders (`db/migrations/001_init.sql:114`) на
  `order_tracking_numbers` (`trg_tracks_audit`, там же `:140`) пишет `to_jsonb(NEW)`
  в `order_change_log.new_row`. Значит `source`/`added_by`/`added_at` попадают в
  журнал автоматически.

**GUC `app.source` НЕ используем для треков из delivery.** Причины:
1. `SET LOCAL app.source` ставится на сессии delivery, а `postgres_fdw` открывает
   отдельную удалённую сессию — GUC границу FDW не пересекает, на стороне
   ebay_orders `current_setting('app.source')` будет пустым.
2. `change_source` — это ENUM `('screenshot','user_chat')`
   (`001_init.sql:10`); постороннего значения он не примет, а расширять его смысла
   нет — источник уже лежит в колонке `source`.

Поэтому `order_change_log.source` для FDW-записей останется пустым; «кто/что»
читается из `new_row` (jsonb) и из колонок. Enum `change_source` не трогаем.

## 5. Изменения в `delivery`

### 5.1. FDW — `sql/002_fdw.sql`

- Пересоздать FOREIGN TABLE `ebay_remote.order_tracking_numbers` с колонками
  `source text`, `added_by text`, `added_at timestamptz` — без них INSERT через
  FDW не сможет проставить источник.
- Заодно фикс мины: `ebay_remote.order_items` объявлена с колонкой `quantity int`
  (`002_fdw.sql:56`), а реальная колонка в ebay_orders — `item_quantity`
  (`ebay_orders/db/migrations/001_init.sql:36`). View берёт только `item_title`,
  поэтому сейчас не падает, но любой `SELECT quantity` через FDW сломается.
  Переименовать в `item_quantity` либо убрать (не используется).

### 5.2. View `parcels` — `sql/004_fdw_parcels_aggregate.sql`

Драйвер строк остаётся `FROM parcels_mutations pm LEFT JOIN agg` — поэтому деньги/
вес **не задваиваются**: посылка существует только там, где есть строка
`parcels_mutations` (то есть на каноне), а алиас своей строки не имеет.

Что добавить во view:
- Поле `track_numbers text[]` — все номера группы (канон + алиасы). Считать в CTE
  `agg`, не построчным коррелированным подзапросом (это N round-trip'ов по FDW).
- Алгоритм `agg`: для каждого канонического трека собрать заказы группы, по ним
  один раз посчитать `sold_by` (`single_or_raise`), `SUM(order_total_usd)`,
  `MAX(delivered_date)`, `STRING_AGG(item_title)` и `array_agg(DISTINCT tracking_number)`
  всех треков этих заказов.

Осторожно: `single_or_raise` (`004:16`) бросает `EXCEPTION` при конфликте
продавцов в группе — это hard-fail на **весь** `SELECT * FROM parcels`, не на одну
строку. По домену конфликт невозможен (один трек у разных продавцов не бывает), но
помнить про это поведение.

### 5.3. Pool / DSN

- Отдельный pool на ebay_orders **не заводим** — пишем через FDW из основного
  `delivery_pg_dsn`. На одну переменную меньше.
- Pool `delivery_agent` — как и раньше, для заявок и AI-чата (продуктовая спека).

### 5.4. `parcels_mutations`

- Ключ — `tracking_number` = канонический трек. Не меняется.
- `source_order_number` остаётся `NOT NULL`. У бес-канонной группы (см. §1.3)
  delivery подставляет один из `order_number` группы.
- `is_manual` не добавляем (источник в колонке `order_tracking_numbers.source`).

## 6. Алгоритмы (словами)

### 6.1. Резолвер «любой номер → канонический трек»

Единственная функция, через которую проходят все обращения к посылке по номеру.
Вход — произвольный трек (канон или алиас). Шаги:

1. Найти все `order_id`, к которым привязан этот трек (через FDW
   `ebay_remote.order_tracking_numbers`).
2. Среди всех треков этих заказов найти тот, по которому есть строка
   `parcels_mutations` — это канон. Вернуть его.
3. Если строки `parcels_mutations` нет ни по одному треку группы — посылки ещё
   нет (трек незнакомый/непривязанный) → вернуть «не найдено».

Через резолвер обязаны проходить: поиск, `GET /parcels/{tn}`, `PATCH /parcels/{tn}`,
добавление/удаление из отправки, приём отправки (см. §6.4 — иначе тихий no-op).

### 6.2. Привязка трека из delivery (accept плашки)

Вся операция — одна транзакция из основного pool, запись в ebay_orders через FDW.

1. Определить группу: по выбранному админом/AI совпадению (трек или заказ) найти
   полный набор `order_id` группы.
2. Взять advisory-lock по идентификатору группы (например, `min(order_id)`), чтобы
   исключить параллельное создание двух посылок на одну группу (см. §7-E1).
3. Снять `is_untracked`: `UPDATE ebay_remote.orders SET is_untracked=false
   WHERE order_id = ANY($1) AND is_untracked` — иначе `DEFERRABLE`-триггер
   `trg_tracks_invariants` (`ebay_orders/.../001_init.sql:222`) уронит транзакцию
   на удалённом commit.
4. Идемпотентный INSERT треков: для каждого `order_id` группы вставить
   `(order_id, T_new, source='delivery', added_by=логин)`. **Без `ON CONFLICT`** —
   `postgres_fdw` его на foreign-таблице не поддерживает (см. §7-D1); идемпотентность
   обеспечить пред-`SELECT`'ом (трек уже привязан → пропустить).
5. Определить, есть ли у группы канон с посылкой:
   - **Есть** (по любому треку группы существует `parcels_mutations`): новой
     посылки не создаём — `T_new` стал алиасом. Готово.
   - **Нет** (бес-канонная группа, бывший `is_untracked`): delivery сам создаёт
     `parcels_mutations` по `T_new` (он становится каноном), `source_order_number`
     = один из `order_number` группы, `status='ordered'`, `ordered_at`/`eta_usa`
     как в sync (§6.3).
6. Любая ошибка на шагах 3–5 → откат всей транзакции целиком.

### 6.3. Sync — `sync/main.py`

Назначение: заводить `parcels_mutations` для новых eBay-треков.

- `SQL_FETCH_NEW` берёт только `source='ebay'` (через FDW). Это:
  - чинит текущую регрессию (запрос обращается к `o.delivery_extra_tracks`,
    которой больше нет — sync падает каждые 5 минут и молча глотает ошибку);
  - не даёт sync задублировать посылку на алиасе: алиас `source='delivery'` в
    выборку не попадает (иначе `LEFT JOIN parcels_mutations ... IS NULL` счёл бы
    его новым и создал вторую посылку — см. §7-C1).
- Бес-канонные группы (только `source='delivery'`) sync **не трогает** — их
  посылку создаёт сам delivery при привязке (§6.2 шаг 5). Это же закрывает гонку
  delivery↔sync: пересоздать с другой стороны нечем.
- Решить судьбу `raise` в цикле (`main_loop`): сейчас его убрали, падения скрыты.
  Вернуть `raise` (контейнер рестартует, видно в логах) либо оставить тихий лог —
  поведение зафиксировать сознательно.

### 6.4. Приём, отправки, поиск (роуты delivery)

Эти места ключуются по `tracking_number = canonical` и по алиасу молча промахнутся —
все обязаны идти через резолвер (§6.1):

- **Поиск** (`api/app/routes/parcels.py:77,80`) — искать по элементам
  `track_numbers[]`, а не только по `p.tracking_number`. Иначе форвардер,
  отсканировав физический (алиасный) номер, посылку не найдёт.
- **`GET` / `PATCH /parcels/{tn}`** (`:114`, `:134`) — резолвить вход в канон.
- **Отправки** `POST/DELETE /shipments/{sid}/parcels` (`shipments.py:156,193`) —
  резолвить вход в канон.
- **Приём** `receive_shipment` (`shipments.py:307`) — **самое опасное**:
  `UPDATE parcels_mutations ... WHERE tracking_number=$1` по алиасу обновит 0 строк
  **без ошибки** (посылку «приняли», статус не сменился, никто не заметил).
  Обязательно резолвить вход в канон.

## 7. Подводные камни (свод)

- **A1.** `single_or_raise` роняет весь `SELECT parcels`, а не одну строку — см. §5.2.
- **A2.** `track_numbers[]` считать в CTE, а не коррелированным FDW-подзапросом — §5.2.
- **B (резолв).** Шесть точек входа ключуются по канону и промахиваются по алиасу —
  все через резолвер; `receive` промахивается **молча** — §6.4.
- **C1.** Sync задублирует посылку на алиасе, если не фильтровать `source='ebay'` — §6.3.
- **C2.** Бес-канонную группу sync не заведёт — посылку создаёт delivery сам — §6.2/§6.3.
- **D1.** `ON CONFLICT` через FDW не поддерживается — идемпотентность пред-`SELECT`'ом — §6.2.
- **D2.** UPDATE `is_untracked` + INSERT треков — строго одной FDW-транзакцией,
  иначе `DEFERRABLE`-инвариант уронит на commit — §6.2.
- **E1.** Параллельная привязка к бес-канонной группе → две посылки; защита —
  advisory-lock по группе — §6.2.
- **E2.** Гонка delivery↔sync закрыта тем, что sync игнорирует `source='delivery'` — §6.3.
- **Конфликт «трек ↔ продавец» НЕ проверяем** в delivery — по домену такого кейса
  нет (один трек у разных продавцов невозможен). Валидатор в ebay_orders
  (`tools.py:731`) остаётся для пути `save_order`, но через FDW он не вызывается и
  дублировать его в delivery не нужно.

## 9. Что фиксируем в `SPEC_UNKNOWN_TRACKS.md` (потом)

При доработке продуктовой части синхронизировать формулировки:
- «Где что хранится»: убрать `delivery_extra_tracks`/`_meta`; везде
  `order_tracking_numbers` с `source`.
- «Принятие плашки»: сослаться на §6.2 этой спеки за алгоритмом привязки.
- «Аудит»: запись нашего трека живёт в `ebay_orders.order_change_log.new_row`
  (jsonb с `source='delivery'`/`added_by`) + параллельный аудит решения в
  `lookup_*`-таблицах delivery_agent.

Сейчас формулировки в продуктовой спеке не правим, чтобы не вести одну мысль из
двух мест.

## 10. Что НЕ делаем сейчас

- **Полная M2M-нормализация** `tracks(id) + order_tracks(...)`. Текущая
  `order_tracking_numbers(order_id, tracking_number)` уже даёт многие-ко-многим;
  модель «канон + алиасы» закрывает потребность без отдельной таблицы `tracks`.
  Нормализация — отдельный заход «на потом» (нужна, в т.ч. для глобальной identity
  трека в `central_smart_logic`, где `tracking_number` используется как `owner_id`),
  переписывать пришлось бы ebay_orders + FDW + sync разом.
- **UI треков в ebay_orders** — не нужен, всё через чат-агента.
- **Форвардер подтверждает привязку без админа** — флаг настройки добавим позже
  без правок схемы.
